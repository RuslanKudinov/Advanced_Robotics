
# Домашнее Здание 1: Обработка данных лидара
## Wall Detection (2D) + Ground Detection (3D)

### Описание проекта
В рамках данной работы реализованы два классических алгоритма обработки данных лидара: детектирование стен в 2D-скане с помощью преобразования Хафа (Hough Transform) и выделение плоскости пола в 3D-облаке точек с помощью RANSAC.

---

## Задание 1. Wall Detection (2D LIDAR + Hough Transform)

### Цель
Найти прямые линии (стены) в облаке точек 2D-лидара.

### Данные
Используется синтетический датасет — массив точек, образующий прямоугольник (контуры комнаты) с добавлением гауссовского шума. Параметры комнаты: форма — сложный контур с выступами, размеры от 0 до 12 метров по X, от 0 до 10 метров по Y, шаг дискретизации стен 0.4 метра, уровень шума σ = 0.1 метра.

### Реализация
**Генерация синтетических данных:**
```python
vertex = np.array([[0, 0], [0, 10], [8, 10], [8, 7], [12, 7], [12, 0], [8, 0], [8, 2], [4, 2], [4, 0]])
walls = make_walls(vertex, body=np.array([5, 7]), step=0.4, scale=0.1)
```

**Преобразование Хафа:**
- Перевод координат точек в пространство параметров (θ, ρ)
- Дискретизация углов: 360 шагов от -90° до 90°
- Построение аккумуляторной матрицы
- Поиск пиков с порогом 10% от максимума

**Визуализация:**
- Исходное облако точек (синие точки)
- Найденные линии стен (красные линии)
- Истинные углы комнаты (зеленые кресты)

### Результаты задания 1
Алгоритм Split and Merge (Ramer-Douglas-Peucker) применен для выделения ключевых точек контура: было точек 121, стало ключевых точек 11 с точностью аппроксимации ε = 0.5.

## Задание 2. Ground Detection (3D + RANSAC)

### Цель
Выделить плоскость пола из 3D облака точек, чтобы определить зоны, пригодные для навигации робота.

### Данные
Используется пример из библиотеки Open3D (PCDPointCloud). Для ускорения обработки облако точек прореживается в 10 раз. Источник: Open3D example dataset, количество точек после даунсемплинга 11 367, тип данных — 3D координаты (X, Y, Z) в метрах.

### Реализация RANSAC
Алгоритм итеративно ищет наилучшую плоскость. Для каждой итерации (max_trials = 1000) случайно выбираются 3 точки, строится плоскость ax + by + cz + d = 0, вычисляются расстояния от всех точек до плоскости, подсчитывается количество инлайеров (|distance| < 0.03) и сохраняется лучшая модель.

Параметры: порог расстояния 0.03 м, количество итераций 1000, минимальное число точек для плоскости 3.

### Результаты задания 2

| Тип точек | Количество | Процент |
|-----------|------------|---------|
| Инлайнеры (пол) | 8 793 | 77.4% |
| Аутлайнеры (препятствия) | 2 574 | 22.6% |
| Всего | 11 367 | 100% |

Визуализация: зеленые точки — инлайнеры (плоскость пола/стола), красные точки — аутлайнеры (препятствия, стены, объекты).

Анализ чувствительности параметров: при уменьшении количества итераций до 10 результат становится нестабильным (от 7 292 до 8 868 инлайнеров). Для надежного обнаружения плоскости необходимо не менее 1000 итераций.



## Технологии
Python 3.x, NumPy, Matplotlib, Open3D, Plotly, Scikit-image, Random.



## Выводы по работе

**По заданию 1 (Hough Transform):** преобразование Хафа успешно выделяет основные стены помещения даже при наличии шума в данных, алгоритм устойчив к пропускам точек и локальным выбросам. Split and Merge (RDP) позволяет значительно сократить количество точек при сохранении формы контура.

**По заданию 2 (RANSAC):** RANSAC эффективно выделяет доминирующую плоскость в зашумленных 3D-данных, алгоритм устойчив к выбросам — плоскость пола найдена даже при наличии множества объектов. Количество итераций критически влияет на стабильность результата.

**Общие выводы:** оба алгоритма являются фундаментальными инструментами в робототехнике и компьютерном зрении. Преобразование Хафа позволяет роботу "видеть" стены и ориентироваться в помещении, а RANSAC — определять безопасные зоны для движения.

# Домашнее задание 3: Сравнительный анализ EKF для оценки ориентации
## Сравнительный анализ EKF для оценки ориентации (Углы Эйлера vs Кватернионы)

### 📌 Описание проекта
Реализация и сравнение двух подходов к расширенному фильтру Калмана (EKF) для оценки ориентации смартфона в пространстве:
- **Вариант A**: EKF на основе углов Эйлера (Roll, Pitch, Yaw)
- **Вариант B**: EKF на основе кватернионов

### 📊 Данные
- Запись с акселерометра и гироскопа смартфона (668 отсчетов)
- Протокол: покой → вращение по всем осям → критический тест (Pitch = 90°) → возврат
- Формат CSV: `time, ax, ay, az, wx, wy, wz`

### 🧠 Реализация

#### Фильтр A (Euler-based EKF)
- Вектор состояния: `[φ, θ, ψ]ᵀ`
- Prediction: интегрирование угловых скоростей через матрицу перехода
- Update: проекция вектора гравитации через матрицу поворота

#### Фильтр B (Quaternion-based EKF)
- Вектор состояния: `[qw, qx, qy, qz]ᵀ`
- Prediction: кинематическое уравнение `q̇ = ½ q ⊗ ω` + нормализация
- Update: проекция гравитации через кватернион + коррекция

### 📈 Результаты

#### 1. Сравнение углов ориентации
На графиках видно, что до определенного момента оба фильтра выдают идентичные значения Roll, Pitch и Yaw. Это подтверждает корректность реализации.

#### 2. Gimbal Lock (Критический тест)
При приближении угла Pitch к **90 градусам**:
- **Эйлеров фильтр** теряет устойчивость — значения углов уходят в `10¹¹°` из-за сингулярности матрицы перехода (`cos(θ) → 0`)
- **Кватернионный фильтр** продолжает корректно работать, плавно проходя через критическую точку

#### 3. Анализ нормы кватерниона
- Без нормализации норма уплывает от единицы из-за ошибок интегрирования
- После нормализации на каждом шаге норма возвращается к `1.0`, обеспечивая корректность вычислений

### ✅ Выводы

|---------|-------------|-------------|

**Итог:** Кватернионный подход предпочтительнее для задач навигации и оценки ориентации, особенно при работе с большими углами. Несмотря на бóльшую математическую сложность, он обеспечивает глобальную неособенную параметризацию и высокую численную устойчивость.
